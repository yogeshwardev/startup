CampusArena Admin and Teacher Pages - Detailed Feature List

This document describes the currently implemented teacher-side and admin-side pages in the project, based on the frontend routes and the connected backend APIs.

==================================================
1. TEACHER PAGES
==================================================

Teacher access is available for users with role TEACHER.
Some teacher routes also allow ADMIN access for oversight/testing, but they are part of the teacher workflow.

----------------------------------------
1.1 Teacher Dashboard
Route: /teacher
Component: client/src/pages/teacher/TeacherDashboard.jsx
----------------------------------------

Main purpose:
- Gives teachers a quick overview of student readiness and visibility.

Implemented features:
- Loads student data through the user listing API with role=STUDENT.
- Shows summary cards for:
  - Students in scope
  - Active students
  - Years covered
- Defines active students as students whose accounts are not blocked.
- Displays a "Recently visible students" section.
- Shows up to 6 student cards with:
  - Student name
  - Department
  - Year
- Uses loading skeletons before data is available.

Behavior notes:
- Teacher visibility is department-scoped by backend rules when the logged-in user is a teacher.
- The dashboard is read-focused and does not provide editing directly.

----------------------------------------
1.2 Teacher User Management / Student Roster
Route: /teacher/users
Component: client/src/pages/admin/UserManagementPage.jsx
----------------------------------------

Main purpose:
- Lets teachers review students in their scope and make limited profile/status changes.

Implemented features:
- Reuses the same user management page as admin, but in teacher mode.
- Loads visible users from /users.
- Backend restricts teacher results to:
  - role = STUDENT
  - same department as the teacher
- Shows summary cards for:
  - Total visible users
  - Selected rows
  - Departments
  - Active visible students
- Provides user table with:
  - Checkbox selection
  - Name
  - Email
  - Role badge
  - Department
  - Year
  - Status badge
  - Action buttons
- Supports filtering by:
  - Search text
  - Role
  - Department
  - Year
- Supports sorting by:
  - Name
  - Email
  - Role
  - Department
  - Year
  - Status
- Supports pagination with previous and next controls.
- Allows row selection for quick review.
- Allows refresh of the data table.
- Provides action buttons per row:
  - View profile
  - Edit user
  - Toggle active/blocked status

Teacher limitations:
- Teachers cannot create users.
- Teachers cannot delete users.
- Teachers cannot bulk delete users.
- Teachers cannot reset passwords.
- Teachers cannot edit admin or teacher accounts.
- Teachers can only edit students from their own department.
- Teachers cannot change permissions.
- Teachers cannot change a student role because role edits are admin-only in backend logic.

Editable teacher-side fields in the modal:
- Name
- Email
- Registration number
- Department
- Year
- Active/blocked state

Important backend rule:
- Teacher updates are allowed only for student accounts in the teacher's department.

1.4 Teacher Problem Assignment Page
Route: /teacher/problems
Component: client/src/pages/teacher/TeacherProblemsPage.jsx
----------------------------------------

Main purpose:
- Allows teachers to assign practice problems to students.

Implemented features:
- Loads three data sets in parallel:
  - All coding problems from /problems
  - Visible students from /students
  - Existing teacher assignments from /teacher/assignments
- Provides an assignment form with:
  - Student selector
  - Problem selector
  - Optional due date/time
- Problem selector shows:
  - Problem title
  - 6-digit problem ID
  - Difficulty
- Automatically preselects the first available student and first available problem after load.
- Submits assignments through POST /teacher/assignments.
- Shows toast feedback for success and failure.
- Refreshes assignment list after creating a new assignment.
- Displays existing assignments with:
  - Problem title
  - Problem ID
  - Student name
  - Student department
  - Assignment status
- Shows empty state if no assignments exist.

Practical use:
- Helpful for directed practice and deadline-based preparation.

1.5 Teacher Mock Test Management
Route: /teacher/mock-tests
Component: client/src/pages/teacher/TeacherMockTestPage.jsx
----------------------------------------

Main purpose:
- Lets teachers create and manage mock tests for students.

Implemented features:
- Loads all coding problems from /problems.
- Loads existing mock tests from /admin/mock-tests.
- Mock test creation form includes:
  - Mock test title
  - Duration in minutes
  - Optional company/topic label

  - Search problems by title
  - Search by problem ID
  - Search by difficulty

- Prevents creation if no problem is selected.

  - Order number
  - Problem title
  - Problem ID
  - Difficulty
- Displays created mock tests with:
  - Title
  - Duration
  - Number of problems

- Shows loading, saving, and deleting states.
- Shows empty states for:
  - No selected problems
  - No mock tests

Notes:
- Teachers are allowed by backend authorization to manage mock tests.
- This page is one of the teacher portal's strongest creation tools.

2.3 Admin Create User Page
Route: /admin/users/new
Component: client/src/pages/admin/UserCreatePage.jsx
----------------------------------------

Main purpose:
- Create one user manually or many users in bulk.

Implemented features:
- Loads departments for selection.
- Manual user creation form includes:
  - College email
  - Registration number
  - Temporary password
  - Department
  - Year
  - Role
- Can auto-generate a strong password.
- Validates academic email pattern.
- Validates registration number presence.
- Validates strong password rules.
- Submits manual creation through POST /users.
- Redirects back to /admin/users after success.

Bulk creation features:
- Accepts CSV upload.
- Also supports pasted CSV text through internal state flow.
- Provides downloadable sample CSV file:
  - /samples/bulk-users-template.csv
- Parses rows in this format:
  - email, registrationNumber, password, department, year, role
- Validates each row before upload.
- Shows preview of first few parsed rows.
- Submits bulk creation through POST /users/bulk.
- Displays created user count on success.

Backend validation notes:
- Department must already exist.
- Duplicate emails are rejected.
- Duplicate registration numbers are rejected.
- Only admins can create or bulk-create users.

==================================================
3. SHARED ROLE-BASED BEHAVIOR
==================================================

These behaviors are important across both teacher and admin areas.

Authentication and access:
- /teacher/* routes are protected for TEACHER and ADMIN.
- /admin/* routes are protected for ADMIN only in the frontend router.
- Most management APIs are protected with backend authorization.

Navigation:
- Teacher navigation includes:
  - Dashboard
  - Profile
  - Users
  - Progress
  - Departments
  - Problems
  - Mock Tests
  - Settings
- Admin navigation includes:
  - Dashboard
  - Profile
  - Users
  - Departments
  - Problems
  - Existing Problems
  - Contests
  - Mock Tests
  - Placement Companies
  - Placement Questions
  - Analytics
  - Settings

Theme and shell features:
- Both admin and teacher pages use the same application shell.
- Top bar shows role-specific workspace label:
  - Control Center for admin
  - Faculty Desk for teacher
- Both can toggle dark/light theme.
- Both can view profile from the top bar.
- Both can sign out from the profile menu.

Data and workflow conventions:
- Problem management relies on reportable 6-digit problem IDs.
- Department names are normalized to uppercase in backend logic.
- Teacher access to students is department-scoped.
- Admin user actions generate audit activity logs.
- Mock tests support reusable problem selection flows.

==================================================
4. IMPORTANT IMPLEMENTATION NOTES
==================================================

These notes reflect the current codebase behavior.

- Teacher user management is intentionally limited compared to admin user management.
- Teacher analytics are lightweight and derived from visible student lists, not full platform analytics.
- Teacher mock test management currently uses the admin mock test endpoints, but the backend allows teachers to use them.
- Admin problem creation and dedicated contest management both exist, so contest creation appears in two places.
- The admin question modal prepares constraints in payload, but the visible form currently does not expose a dedicated constraints textarea in the UI.
- The sidebar contains a teacher navigation item for /teacher/students labeled "Progress", but the route defined in App.jsx is /teacher/users rather than /teacher/students.

End of document.
